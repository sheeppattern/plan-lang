# Plan Language란 무엇인가

## 기존 기획의 문제

소프트웨어를 기획할 때 우리는 PRD(제품 요구사항 문서), Jira 티켓, Confluence 페이지, 슬랙 메시지 등 다양한 도구를 사용한다. 이 도구들에는 공통된 한계가 있다:

**"아직 모르는 것"을 다루지 못한다.**

기획 초기에는 결정되지 않은 것이 많다. PG사가 아직 선정되지 않았거나, 할인율이 미정이거나, 기술적 가정이 검증되지 않았을 수 있다. 기존 도구에서는 이런 불확실성을 주석이나 관례로만 표현한다. "TBD", "(확인 필요)", "TODO" 같은 표시를 다는 것이다. 하지만 이런 표시는:

- 기계가 파싱할 수 없다 (검색도, 집계도, 자동 검증도 불가능)
- 유형 구분이 없다 ("미정"과 "가정"과 "위험"이 모두 같은 "TBD")
- 빠뜨려도 알 수 없다 (불확실한 것을 표시하는 걸 잊어버리면 그냥 확정된 것처럼 보인다)

## Plan Language의 접근

Plan Language는 **불확실성을 포맷의 1급 시민으로** 다룬다.

`.plan` 파일은 Markdown과 완전 호환되는 텍스트 파일이다. 줄 시작에 키워드(`Goal:`, `When:`, `Then:` 등)를 붙여 기획의 구조를 잡고, `?pending`, `?assumption` 같은 불확실성 마커로 "아직 모르는 것"을 명시적으로 표현한다.

핵심 아이디어는 단순하다:

1. **구조화된 기획** — 12개 키워드로 기획을 Feature → Story → Task 3단계로 분해한다
2. **불확실성 추적** — `?pending`이 남아 있으면 기획이 "준비 완료" 상태로 전이할 수 없다
3. **Markdown 호환** — `.plan` 파일을 GitHub, VS Code, Notion에서 열면 그냥 Markdown으로 읽힌다
4. **기계 파싱 가능** — 파서, 린터, CLI, LSP 서버로 자동 검증한다

## 누가 사용하는가

Plan Language는 세 종류의 사용자를 위해 설계되었다:

| 사용자 | 사용 방식 |
|--------|----------|
| **PM / 기획자** | `.plan` 파일로 기획을 작성하고, 린터로 완성도를 검증한다 |
| **개발자** | 기획의 수용 기준(Given-When-Then)과 Task를 읽고 구현한다 |
| **AI 에이전트** | Task 블록을 JSON Schema로 변환받아 자동 구현하고, Verify 조건으로 검증한다 |

## 다른 도구와의 비교

| | PRD (문서) | Jira (이슈 트래커) | Gherkin (BDD) | Plan Language |
|---|---|---|---|---|
| 불확실성 표현 | 자유 텍스트 | 없음 | 없음 | `?pending`, `?assumption` 등 4종 |
| 기계 파싱 | 불가 | API 의존 | 가능 | 가능 |
| Markdown 호환 | 부분적 | 불가 | 불가 | 완전 호환 |
| AI 에이전트 지원 | 제한적 | 제한적 | 제한적 | Task → JSON Schema 변환 |
| 상태 검증 | 수동 | 워크플로우 | 없음 | `?pending` → `ready` 전이 차단 |

## 설계 원칙

Plan Language는 검증된 패턴들을 조합하여 만들었다:

- **Gherkin (BDD)** — Given-When-Then 키워드로 수용 기준을 기술하는 패턴
- **EARS (Rolls-Royce)** — 줄 시작의 키워드로 자연어에 구조를 부여하는 패턴
- **RFC 2119** — MUST/SHOULD/MAY로 의무 수준을 명시하는 패턴
- **YAML 프론트매터** — 파일 상단에 메타데이터를 배치하는 패턴 (Jekyll, Hugo, Cursor)

이 패턴들은 각각 수년~수십년간 실전에서 검증되었다. Plan Language는 이들을 하나의 `.plan` 파일 포맷으로 통합한 것이다.

## 토큰 효율성

AI 에이전트와 함께 사용할 때, `.plan` 파일은 같은 내용을 JSON이나 XML로 표현하는 것보다 훨씬 적은 토큰을 사용한다:

- JSON 대비 **34~38% 절약**
- XML 대비 **약 45% 절약**

Markdown 기반이라 사람이 읽기에도 자연스럽고, AI가 파싱하기에도 효율적이다.
